How this site works

This page is a friendly front-end for a small research agent that hunts for upcoming live music listings. The heavy lifting is done in Python; large language models (LLMs) help plan searches and turn messy web pages into neat rows you can browse here.

🎛️ The pipeline in plain English

  1. Plan. An LLM suggests search queries from your topic configuration (venues, locations, date window — all driven by config, not hard-coded in the app).
  2. Search & crawl. The agent runs those queries, fetches candidate pages, and extracts text (and clues like images) for the next step.
  3. Curate. Another structured LLM pass reads the noisy text and fills in a spreadsheet-style row: act, venue, date, link, short summary, poster URL, and more. The goal is structured data you can trust, not a wall of prose.
  4. Merge & de-duplicate. Deterministic code merges new finds with the on-disk spreadsheet, drops past gigs, and folds duplicate listings so you don’t see the same show three times.
  5. Enrich & cache. Thumbnails are scored and cached locally when possible, so your browser loads posters from this site instead of fragile hotlinks.
  6. Publish. The same database powers this Angular list (via events.json), optional HTML, and other exports.

🤖 Where LLMs fit (and where they don’t)

LLMs are great at turning unstructured blurbs into fields and at brainstorming queries — they are not a database of facts. This project keeps them on a short leash: outputs are validated, dates are checked against a window, and persistence is always the spreadsheet/JSON, not “whatever the model said last.”

You can run against cloud OpenAI or a local OpenAI-compatible stack; the UI you’re reading doesn’t need API keys — it only reads the published snapshot.

🎸 What you’re looking at

The event list is a read-only view of the latest merged data. Refresh or re-run the pipeline when organisers post new shows — the agent does the busywork so this page stays current without manual copy-paste.
