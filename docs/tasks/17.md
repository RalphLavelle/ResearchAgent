# App is slow!
At the moment, probably as a result of some recent changes in the events schema linking them to venues, the main list that shows on the the front page is taking 40+ seconds to load! The site is unusable as a result.

# Suggestion
Can you investigate the link between the mongodb events collection schema and the venues collection - do I need more indexes? Do I need to do two separate lookups - one for the events, then a second, with all the relevant venues to get their location, and update the response object before returning it over the wire?

Investigate and come up with suggestions for:
* what you think is causing the slowness
* the best way(s) to fix it

---

## Investigation (completed)

### Root cause — not missing venue indexes

The venue link itself is fine: events store a nested `venue: { name, id }` subdocument, and location is hydrated from the `venues` collection via **one** batched query (`locations_by_id`). That part is O(venues) and cheap.

The 40+ second load was caused by **how `GET /api/{db}/events` loaded data**:

1. **Triple full scan of the `events` collection** — `load_existing_rows()` scanned all events; inside that, `source_urls_by_event_id()` scanned all events again; then `get_events()` scanned all events a **third** time for `image_id` thumbnail URLs.
2. **N+1 image lookups** — `source_urls_by_event_id()` called `images.find_one()` **once per event** with an `image_id`. With hundreds of events and Atlas latency (~50–100 ms per round trip), that alone explains 40+ seconds.
3. **Pipeline-oriented loader on the hot path** — `load_existing_rows()` is built for merge/dedupe (spreadsheet rows, poster URL resolution for quality scoring). The API only needs a single projection + venue location map.

Indexes on `venue.id` would help admin endpoints like `count_events_for_venue`, but they would **not** fix the main list — that endpoint already does a full collection read.

### Fix applied

- **`load_events_api_payload()`** — dedicated API loader: one events scan + one venues scan; thumbnail URLs built from each event's `image_id` without touching the `images` collection.
- **`source_urls_by_image_ids()`** — batch `$in` query for pipeline code paths that still need upstream poster URLs.
- **Indexes** (lazy, idempotent): `events.venue.id`, `events.date`, `images.source_url` — helpful for admin and poster registration, not the primary fix.

### If the list grows much larger

Consider pagination or projection-only fields on `GET /api/{db}/events`; denormalise `location` onto event documents at write time to skip the venues lookup entirely; or MongoDB `$lookup` aggregation if you prefer a single query (usually unnecessary at current scale once N+1 is removed).
