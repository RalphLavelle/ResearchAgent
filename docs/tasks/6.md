# Dedupe the venues
Some of the venues are showing in the app as different labels even though they represent the same actual venue. For instance, for one event, the venue shows as 
'The Tivoli Theatre' whereas for another it's just 'The Tivoli'. Elsewhere it may be just 'Tivoli'. This is wrong since these are clearly the same venue.

## Venues collection
What's needed is a 'venues' collection, where each venue has a 'name' property and an 'aliases' property (of strings). That way the canonical name could be 'The Tivoli Theatre', but the aliases would allow for 'Tivoli', 'The Tivoli', etc.

At the moment of collation of new events in the database, the venues collection could be checked to see if the venue the app thinks the event is at is already represented, either by name or aliases. If so, link to that existing venue in the events table. If not, create a new record in the venues collection with the name vanue as the name. Venues will have their aliases manually checked from time to time. 

# Migration
When this collection has been created, please go thorugh the existing events and migrate all the existing events venues to a link to the event collection record, but also keep the canonical name (so the venue property becomes a 'name' (string), and a ref to the id of the venue document in the venues collection) to avoid joins when displaying.
