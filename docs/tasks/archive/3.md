# Deduplication idea
There is a module in the agent layer that deals with the problem of duplicate entries. It looks at events it has found and decides whether the event in question is really the same as one we already know about.

## Problem
I think this is a bit limited, because in this case:

* Dead of Winter Festival - 2026 BAND COMPETITION | A band competition as part of the Dead of Winter Festival. | Mo's Desert Clubhouse, Burleigh Heads | Sun 24 May 2026
* Dead of Winter Festival Band Competition | Band competition as part of the Dead of Winter Festival. | Burleigh, QLD | Sun 24 May 2026

we are clearly talking about the same event. The problem arises because all three fields that we might care to compare them on are just different enough to confuse the algorithm into thinking they're distinct.

## Suggestion
Add an id to the list of event properties. This will be a 'hidden' property, not one intended to be shown to the end user in the frontend, but a property to enable the app to unambiguously identify events in the database. 

Leave the existing deduplication logic alone - it doesn't use AI (call an LLM) and so is free, insofar as no tokens are spent using it. However, not using AI may be limiting because I think that an AI (LLM) call is the only way to semantically understand if any two events in the list are really just the same.

Make a new LLM call, sending in all the events, with their property names, e.g. 'name', 'description', etc. (to help the AI make sense of them) asking the LLM if it thinks any two events on the same date are really the same event.

## Handling dupes
In the event of a duplicate, the LLM should merge them in a way that maximises the information (especially ifone of them has a photo). The app should then remove the duplicate and update the 'merged' event.
