# Strategy scores
Now there's a strategy_scores collection in the db, how can we begin to use it to have the app 'learn' from each run? Also, how is it different to the sources collection? That does something similar - remembers which urls from past searches yielded events in order to influence future ones. Is it possible to combine them? Think carefully about this.

## Answer

### How can the app begin to learn from `strategy_scores`?

Today the collection is **write-only**: after each run, `report_store.save_run_report` calls `apply_strategy_scores_for_report`, which accumulates counters per entity (`events_added`, `events_seen`, `duplicates`, `zero_yield_runs`, `runs`, `last_seen`) for four kinds: `source_url`, `host`, `venue`, and `query`.

**Learning** means reading those scorecards back into the next run and biasing choices with a deterministic weight function (positive yield minus zero-yield penalty, time-decayed) plus a fixed exploration floor so the agent never over-fits to past winners.

The first behavioural hook (Prompt 3) wires venue scores into targeted venue query selection: venues with strong historical yield, missing `events_link`, stale `last_mined`, or weak future coverage are sampled more often; low-score venues still get occasional coverage.

Later roadmap steps will use the same read-back pattern for multi-seed crawl memory (Prompt 4), planner query/angle feedback (Prompt 5), and admin visibility (Prompt 7).

### How is it different from `sources`?

| | `sources` | `strategy_scores` |
|---|---|---|
| **Scope** | Multi-event listing URLs only (≥2 distinct events per page) | All source URLs, hosts, venues, and queries — including zero-yield negatives |
| **Shape** | One document per host with a nested `urls` array | Flat documents keyed by `kind:key` |
| **Lifecycle** | Active pruning when `runs_contributed > 3 × events_added` | Passive cumulative counters; idempotent per report |
| **Read-back** | **Yes** — `pick_weighted_seed_url` picks one weighted crawl seed each run | **Now starting** — venue query weighting first; crawl/planner hooks later |

`sources` is a narrow, proven closed loop for crawl seeds. `strategy_scores` is a broad telemetry ledger that can eventually inform planner, venue, and crawl decisions — but it does not yet replace the multi-event gate or stale pruning that keeps crawl memory useful.

### Can they be combined?

**Eventually yes**, because `strategy_scores` is a coverage superset for URL/host yield. **Not yet.** Folding `sources` into `strategy_scores` now would risk regressing the working crawl-seed loop (loss of the multi-event gate, stale pruning, and the existing weighted picker). Keep both collections: `sources` continues to drive crawl memory; `strategy_scores` is the general read-back layer for planner and venue selection until `source_url`/`host` scores can provably reproduce the seed picker behaviour.
